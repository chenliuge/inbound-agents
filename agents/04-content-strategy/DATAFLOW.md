# DATAFLOW：数据流转

> 本文件定义 Agent 04 的输入来源、输出去向、产出结构规范，以及三层产出（季度/月度/周度）的节奏与依赖关系。
> 只管"数据长什么样、从哪来、到哪去"，不管"策略怎么做"。

---

## 一、三层产出节奏

| 层级 | 频率 | 稳定性 | 输出池 | 说明 |
|---|---|---|---|---|
| 季度策略 | 每季度 1 次 | **定死不变** | 04-strategy-pool | 季度内不调整，作为月度/周度的不变约束 |
| 月度计划 | 每月 1 次 | **季度框架内可调** | 04-monthly-plan-pool | 根据当月情报/洞察/热点微调重点，不偏离季度方向 |
| 周度选题 | 每周 1 次 | **最灵活** | 05-topic-plan-pool | 根据本周情报/洞察/需求池动态生成 |

**核心原则**：

- **上层约束下层**：周选题必须在月度计划框架内，月度计划必须在季度策略框架内。
- **下层不改上层**：若周选题发现需要调整月度方向，Agent 04 应在产出中标注"建议月度评审点"，但不擅自修改月度计划。
- **季度定死**：季度策略一旦产出，季度内不重写。仅在极端情况（如重大政策变化）由用户明确指令触发重写，Agent 不得自发触发。

## 三、产出路径与触发

### 3.1 触发方式

Agent 04 由**用户对话指令触发**，不自动运行。根据用户请求关键词，只产出**一个层级**，不混合产出。

| 用户请求关键词 | 触发层级 | 输出池 |
|---|---|---|
| "季度策略""Q1/Q2 策略""新一季""战略" | 季度策略 | 04-strategy-pool |
| "月度计划""本月重点""X 月方向""月度内容计划" | 月度计划 | 04-monthly-plan-pool |
| "周选题""本周选题""选题计划""周度内容" | 周度选题 | 05-topic-plan-pool |

若请求不明确，Agent 04 必须先问清楚用户想要哪一层级，不得擅自选择。

### 3.2 上层缺失时的防御

- **生成月度计划时**，若 04-strategy-pool/approved/ 中不存在当季度策略，Agent 04 必须停止产出，提示用户："当季度策略尚未生成，请先生成季度策略再做月度计划。"
- **生成周度选题时**，若 04-monthly-plan-pool/approved/ 中不存在当月计划，Agent 04 必须停止产出，提示用户："当月度计划尚未生成，请先生成月度计划再做周选题。"
- **用户可显式覆盖防御**（如"我知道季度策略没有，先出一版周选题救急"），此时 Agent 04 产出时必须在文首标注"本次产出在无上层约束下完成，建议尽快补齐上层"，并附带一个简化的上层假设作为临时约束。

### 3.3 输出文件命名

由 write_pending 自动生成，格式为：
- 季度策略：`quarterly-strategy-YYYY-Q[X]-时间戳.md`
- 月度计划：`monthly-plan-YYYY-MM-时间戳.md`
- 周度选题：`weekly-topics-YYYY-W[XX]-时间戳.md`

### 3.4 MEMORY 回写

每次产出完成，在产出末尾附"建议写入 MEMORY 的条目"供用户确认，不直接写。典型回写内容：
- 本次产出涉及的选题标题（防重复）
- 本次观察到的账号阶段变化
- 本次观察到的需求池变化规律
- 本次取舍中值得沉淀的经验
- 本次消耗的 demand-pool 条目 ID（用于更新 used_count）

## 五、字段定义

### 5.1 选题表字段取值

| 字段 | 取值范围 |
|---|---|
| 优先级 | 必做 / 可做 / 观望 |
| 归属支柱 | 市场与趋势判断 / 获客方法论 / 案例与数据证据 / 立场与人格 |
| 子方向（仅支柱 2、支柱 4 需填） | 支柱 2：社媒/GEO/KOL/官网/投放；支柱 4：4a/4b/4c/4d |
| 目标平台 | 公众号 / 小红书 / 视频号 |
| 目标决策人 | 老板 / 营销负责人 / 执行 |
| 预期作用 | 吸引 / 专业 / 转化 |
| 来源类型 | 需求池 / 爆款拆解 / 外部热点 / 第三方素材 |
| 需爆款分析 | ✅ 是 / ❌ 否（用于触发 Agent 05 批量分析） |

### 5.2 来源详情字段填法

- **需求池**：标注 demand-pool 中的条目 ID（如 demand-2026-04-0012），同时简要说明原始需求。
- **爆款拆解**：必须包含「参考账号/平台」+「爆款标题」+「结构要素（钩子/观点/形式）」+「嫁接角度」四项。
- **外部热点**：必须包含「热点事件」+「日期/时间窗」+「为什么现在做」三项。
- **第三方素材**（仅支柱 4 的 4c 路径使用）：必须包含「原始来源（访谈/播客/报道链接）」+「发布时间」+「未来流量的解读角度」三项。**不得伪装为本公司一手素材**。

### 5.3 业务价值字段要求

每条选题的业务价值字段必须明确以下两点：

1. **CTA 导向**：默认所有选题 CTA 指向 L0 免费竞争力报告（作为获客钩子）。
2. **L1 触发点**：说明本条选题如何推动读者接近 L1 购买决策。典型 L1 触发点：
   - "让老板意识到自己团队做不来、需要外部支持"
   - "让营销负责人觉得方法论系统化有价值"
   - "让老板看到同类公司已经起跑、自己不行动会掉队"
   - "让读者建立'这家公司真懂我们行业'的认知"
   - "建立长期人格信任，为后续转化铺垫"

业务价值不能只写"吸引流量""建立专业感"这种空话，必须落到 L1 转化的具体触发点上。

```
                    ┌──── 01-intel-pool（情报）
                    │
                    ├──── 02-insight-pool（洞察）
                    │
需求池（跨Agent共享）────┤
demand-pool              │
↑                        │
│ 写入                   │
│                        ▼
01-industry   ──────> 04 Agent ──季度──> 04-strategy-pool（季度定死）    ──┐
02-user-insight                    │                                      │ 约束
03-sales-cs                        ├──月度──> 04-monthly-plan-pool（月内可调）─┐
05-viral-analysis                  │                                         │ 约束
                                   └──周度──> 05-topic-plan-pool（周内灵活）──┘
                                                     │
                                                     ▼
                                    05-viral-analysis / 06-content-creation
```

**三层稳定性**：
- 04-strategy-pool = 季度策略：**季度内定死**
- 04-monthly-plan-pool = 月度计划：**月内可调，不偏离季度**
- 05-topic-plan-pool = 周度选题：**周内灵活，不偏离月度**

**与 05-viral-analysis 的协作**：

当 04 的选题来源类型为"爆款拆解"时：
- 04 负责指出「参考哪个爆款、嫁接什么角度」
- 05 负责「具体拆解爆款结构」
- 05 的拆解结果反哺 06（内容制作）的执行
